# 业务友好追问指南

> verified: 2026-04-16

Stage 1-2 用，按业务语言追问需求空白，禁止用技术术语。

## 黄金法则

1. **不说"外键"，说"它和谁有关系"**
2. **不说"枚举值"，说"有哪些可选项"**
3. **不说"权限矩阵"，说"谁能看/改/删什么"**
4. **不说"状态机"，说"它从开始到结束会经过哪些阶段"**
5. **每个问题给候选答案**——不让用户从零想，最多 4 个候选 + "其他" 选项
6. **每个问题给业务影响**——告诉用户为什么问这个

## 一次问几个？

- 一次问 3-5 个，不要一口气问完
- 优先问会影响后续设计的高优先级问题（关系/状态/权限）
- 简单字段类型（电话/邮箱/日期）可以直接默认，不必问

## 追问模板（按维度分类）

### 实体维度

```
Q: 你这个 [应用名] 主要管理的"东西"有哪些？
（可多选 + 补充）
- A. <从资料推断到的实体1>
- B. <从资料推断到的实体2>
- C. 还有别的，例如：___

业务影响：决定建几张表
```

### 字段维度（高频）

```
Q: [实体名] 的 [字段名] 有哪些可选项？
- A. <最常见的几个>（推荐 ✓）
- B. <另一种常见组合>
- C. 其他：___

业务影响：决定下拉选项里能选什么
```

```
Q: [实体名] 的 [字段名] 是必填还是可选？
- A. 必填——没填不让保存
- B. 可选——可以为空

业务影响：录入时会不会被卡
```

### 关系维度

```
Q: 一个 [实体A] 可以对应几个 [实体B]？
- A. 一个 A 只能有一个 B（一对一）
- B. 一个 A 可以有多个 B（一对多，常见）
- C. 一个 A 可以对应多个 B，一个 B 也可以对应多个 A（多对多）

业务影响：决定怎么连接两张表
```

```
Q: 删除 [实体A] 时，它名下的 [实体B] 怎么处理？
- A. 不允许删除——必须先把 B 处理完（推荐 ✓，避免数据丢失）
- B. 允许删除，B 一起删
- C. 允许删除，B 保留但变成"无主"

业务影响：误删数据的恢复成本
```

### 生命周期维度

```
Q: [实体名] 从产生到结束会经过哪些阶段？
（按顺序列出，可补充）
- A. <推断的阶段1> → <阶段2> → <阶段3>
- B. 还有特殊状态，例如：___（如"已暂停"、"已取消"）

业务影响：决定能按哪些状态分组、做哪些自动化提醒
```

```
Q: 状态可以从 X 跳回 Y 吗？
- A. 不可以，状态只能往前走（推荐 ✓，简单）
- B. 可以，允许回退（要支持"打回"操作）

业务影响：防止误操作 / 灵活性
```

### 角色与权限维度

```
Q: 用这个系统的人有哪几类角色？
- A. <推断的角色1，例如：销售>
- B. <推断的角色2，例如：经理>
- C. <推断的角色3，例如：管理员>
- D. 其他：___

业务影响：决定建几个角色 + 每个角色的权限范围
```

### 🆕 非飞书用户角色识别（2026-04-16 新增，来自银杏家园实测暴露）

**必问**：除了在飞书里用系统的人，还有什么人要与这个系统交互？

```
Q: 是否有「不登录飞书」但要与系统交互的人？
- A. 有：一线员工/合作方/志愿者（只通过表单提交数据）— 推荐走 Form 分享链接
- B. 有：潜在客户/捐赠人/合作方（只通过公开看板看数据+留表单）— 推荐走对外 Dashboard + Form
- C. 有：资助方/上级监管（只看脱敏汇总报告）— 推荐走飞书只读账号 + 字段级权限过滤
- D. 无
```

**关键追问（每个非飞书角色都要问）**：
```
Q: 这个非飞书角色有几条「上行通道」（他们要提交什么数据给系统）？
例如：月度数据 / 请款申请 / 整改反馈 / 素材上传 / 授权书 / 意向登记 ...
每一条上行通道 = 一个 Form

Q: 这个角色有几条「下行通道」（他们要从系统看到什么）？
例如：业务进度 / 通知提醒 / 统计看板 ...
每一条下行通道 = 一个 Dashboard 或分享链接 或 自动推送（外部邮件/短信/微信）

业务影响：Stage 3 schema 的 forms[] 数量 = 所有非飞书角色的所有上行通道之和；漏一条就是漏一个场景
```

**反模式警告**：
- ❌ 把"站点端只通过表单"当单一场景——实际可能有 3-5 条独立的上行通道（月度数据/请款/月报/整改反馈/素材）
- ❌ 把"基金会只读"当简单权限——实际需要定义「脱敏视图」（哪些字段对他可见）+「导出形式」（飞书里看？导 PDF？）

```
Q: [角色名] 能看到哪些数据？
- A. 只能看自己负责/创建的（推荐 ✓，最常见）
- B. 能看自己部门/团队的
- C. 能看全部
- D. 其他：___

业务影响：数据隔离 + 隐私保护
```

```
Q: [角色名] 能修改/删除别人的数据吗？
- A. 不能（只读）
- B. 可以修改但不能删除
- C. 都可以

业务影响：操作权限边界
```

### 自动化维度

```
Q: 什么情况下要自动通知谁？
（可多选 + 补充）
- A. 新建 [实体] 时通知 [角色]
- B. [实体] 状态变成 [状态] 时通知 [角色]
- C. [实体] 长时间无更新时预警
- D. 其他：___

业务影响：决定建哪些工作流
```

```
Q: 通知用什么方式？
- A. 飞书消息推送给个人
- B. 推送到群
- C. 邮件
- D. 多个组合

业务影响：决定 workflow 节点配置
```

### 视图维度

```
Q: 平时打开这个表，最常想看到的是什么？
- A. 我负责的 / 我创建的
- B. 待处理 / 待跟进 (按时间或状态过滤)
- C. 按 [字段] 分组的全景
- D. 其他：___

业务影响：决定建哪些视图，哪个设为默认
```

### 仪表盘维度

```
Q: 管理者最想从这个系统看到什么数据/统计？
（可多选）
- A. 总数 / 本月新增数 / 趋势
- B. 按 [字段] 分布的饼图/柱图
- C. 按 [角色] 排名的工作量
- D. 待处理积压 (warning)
- E. 其他：___

业务影响：决定建哪些仪表盘组件
```

### 表单维度

```
Q: 是否需要让"非系统用户"也能录入数据？
（例如：客户提交申请、外部填问卷）
- A. 不需要——只有内部团队用
- B. 需要——内部用户能填，分享链接给外部人填
- C. 只有外部填，内部审核

业务影响：决定是否建表单
```

```
Q: 表单收集的字段需要全填吗？
- A. 全部必填
- B. 部分必填（请列出哪些）
- C. 全部可选

业务影响：表单字段配置
```

## 追问停止条件

满足以下任一即可结束 Stage 2：

1. 5 个维度（实体/字段/关系/状态/权限）所有 `⚠️ 待确认` 都已收敛
2. 用户明确说"剩下的你帮我推断默认就行"——记录默认推断进 `01_requirement.md`，标 `🤖 agent 默认推断`，方便 Stage 4 提醒用户复核
3. 用户说"你按你理解先做，做完看效果再调"——直接进 Stage 3，但 Stage 4 预览必须明显标出 agent 自决项

## 反模式

- ❌ "请问字段类型用 single_select 还是 multi_select？"（用户听不懂）
- ❌ "需要建立 N:N 关联吗？"（用户听不懂）
- ❌ 一口气抛 20 个问题（用户疲劳）
- ❌ 不给候选答案，让用户从零想（增加用户负担）
- ❌ 不解释业务影响（用户不知道为什么要回答）
