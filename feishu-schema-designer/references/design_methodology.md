# 设计方法论：8 阶段详细动作

> verified: 2026-04-16

本文档是 SKILL.md §「8 阶段工作流」表格的展开版。作为 agent，按本文执行每一步。

## 第一性原理：领域建模法（v2 — 2026-04-16 更新）

设计 base 不是"想到什么字段就建什么字段"。5 维度 + 1 个强制交叉矩阵：

1. **业务实体**——什么"东西"被这个系统管理？（客户/订单/项目/SKU/批次⋯）
2. **实体关系**——这些"东西"之间怎么连？（1:N、N:N、级联）
3. **生命周期**——每个东西从产生到消亡有几个状态？状态间怎么转？
4. **角色与权限**——**含非飞书用户角色**（通过表单 / 公开看板交互）。谁能看/改/删什么？
5. **使用场景**——日常最高频的查询/分析/录入操作长什么样？
6. **⭐ 强制交叉矩阵**——**「角色 × 场景(CRUD-AV) × 生命周期」**，每格必须有对应 schema 组件支撑

5 维度并列审视会漏掉角色 × 场景的交叉缺口（2026-04-16 银杏家园实测：24 个实体推出后漏了 20+ 个角色场景）。**矩阵审视是强制步骤**，不是可选。详见 [`coverage_audit.md`](coverage_audit.md)。

## Stage 0 — 资料消化

**输入**：用户提供的原始资料（PDF / Word / Excel / Markdown / 旧 base 链接 / 自然语言）+ 项目目录路径

**动作**：
1. 创建工作目录：`<project_dir>/feishu-design/<YYYY-MM-DD-HHMMSS>_<slug>/`
2. 用 Read 工具读所有资料（PDF/Word/Excel 用相应工具）
3. 提取 5 维度信息：实体、关系、生命周期、角色、场景
4. 标注「已知」vs「未知」——已知项写出来，未知项明确列出待 Stage 1 推断或 Stage 2 追问

**产出**：`00_resources_summary.md`
```markdown
# 资料消化总结

## 已读资料清单
- <文件1>
- <文件2>

## 提取到的业务实体
- 客户（已知字段：名称/电话/状态；状态值不明）
- ...

## 提取到的关系
- ...

## 提取到的生命周期
- ...

## 提取到的角色与权限
- ...

## 提取到的使用场景
- ...

## 资料覆盖度评估
- ✅ 实体维度：完整
- ⚠️ 状态机：部分（客户状态值列表缺失）
- ❌ 角色权限：完全缺失，Stage 2 必追问
```

## Stage 1 — 方案推理

**输入**：`00_resources_summary.md`

**动作**：
1. **能从资料推断的全部产出**——基于业务常识 + 设计模式库（`design_patterns/`）
2. **不能推断的部分**——明确标 `⚠️ 待用户确认 (Q-#)`，Q 编号对应 `03_open_questions.md`
3. 优先匹配设计模式库中的现成模式（CRM / 项目管理 / 库存），按用户具体业务调整

**产出 1**：`01_requirement.md`（业务规格说明）
```markdown
# 应用名称：<>

## 业务背景
<2-3 句>

## 业务实体
### 客户
- 字段：客户名称（必填）/ 电话 / 邮箱 / 负责人 / 状态 / 备注
- 状态：进行中 / 已签约 / 流失（⚠️ 待确认 Q-1）
- 生命周期：进行中→已签约→流失（不可逆）

### 订单
...

## 关系
- 客户 1:N 订单（删客户时订单怎么处理？⚠️ 待确认 Q-2）
- ...

## 角色与权限
⚠️ 完全待确认（Q-3, Q-4, Q-5）

## 核心使用场景
1. 销售按状态查找待跟进客户
2. ...
```

**产出 2**：`02_schema_draft.json`（draft schema，部分留空）
```json
{
  "tables": [...],
  "relationships": [...],
  "_pending": ["Q-1", "Q-2", "Q-3"]
}
```

**产出 3**：`03_open_questions.md`（业务友好追问）
```markdown
# 待确认问题

## Q-1: 客户的状态都有哪些？
**业务影响**：决定能按哪些状态分组、做哪些自动化提醒。

候选答案（可多选/补充）：
- A. 进行中 / 已签约 / 流失（推荐 ✓）
- B. 新客户 / 接触中 / 谈判中 / 已成交 / 已流失
- C. 其他：___

## Q-2: 删除客户时，他名下的订单怎么办？
**业务影响**：避免误删导致订单孤立。

候选答案：
- A. 不允许删除有订单的客户（推荐 ✓）
- B. 允许删除，订单一起删
- C. 允许删除，订单保留但变成"无主"

## Q-3: 谁能查看客户数据？
...
```

详见 [`business_questioning_guide.md`](business_questioning_guide.md)。

## Stage 2 — 业务追问审批（审批门 1）

**动作**：
1. 把 `03_open_questions.md` 中的问题用对话形式逐个问用户（一次问 3-5 个，不要一口气全问完）
2. 用户回答后立即合并进 `01_requirement.md`，删除对应 `⚠️ 待确认` 标记
3. 若用户回答衍生新问题，追加到 `03_open_questions.md` 继续问
4. 收敛标准：`01_requirement.md` 无 `⚠️` 标记 + 用户说"可以"

**产出**：稳定的 `01_requirement.md`

---

## Stage 3 — Schema 完整化 + 程序化校验

**动作**：
1. 把 `01_requirement.md` 完全翻译成 `04_schema_final.json`，对齐 [`schema_format_contract.md`](schema_format_contract.md) 格式
2. 跑程序化校验（用 jq + bash 验证）：
   - 外键闭合：所有 `relationships[].child_table` 都存在于 `tables[].name`
   - 字段类型白名单：`fields[].type` 必须在 lark-base 支持列表（参 `schema_format_contract.md`）
   - 命名一致性：关联字段命名 ≈ 主表名
   - 公式字段引用：`formula` 字段公式只能引用同表已定义字段
   - 角色权限合法：`roles[].table_rule_map / dashboard_rule_map / docx_rule_map` 引用的表、字段、视图和组件都存在；具体结构以 `role-config.md` 为准
3. 校验失败 → 自修复 → 再校验，最多 3 次
4. 3 次仍失败 → 停下，把 issues 报告给用户，等指示

**产出 1**：`04_schema_final.json`
**产出 2**：`05_validation_report.json`
```json
{
  "status": "pass",
  "checks": [
    {"rule": "foreign_key_integrity", "result": "pass"},
    {"rule": "field_type_whitelist", "result": "pass"},
    {"rule": "naming_consistency", "result": "warning", "issues": [...]}
  ],
  "iterations": 1
}
```

## Stage 4 — 业务效果预览审批（审批门 2）

**动作**：把 `04_schema_final.json` 翻译成 `06_proposal.md`，纯业务语言，**禁止出现 lark-cli 命令**。

**`06_proposal.md` 必含板块**：

```markdown
# <应用名称> 最终方案预览

## 一、整体架构
共 N 张表，X 个角色，Y 个自动化流程，Z 个仪表盘

## 二、表与字段
### 客户表
| 字段 | 类型 | 必填 | 说明 |
| 客户名称 | 单行文本 | 是 | 主字段 |
| 状态 | 单选 | 是 | 进行中/已签约/流失 |
| 负责人 | 人员 | 是 | 关联用户 |
...

### 订单表
...

## 三、表间关系
- 客户 ←→ 订单（一个客户可有多个订单，删客户时订单受保护）

## 四、视图（每张表的常用视图）
### 客户表
- **客户总览**：按状态分组，显示名称/负责人/最近跟进
- **我的待跟进**：仅显示当前用户负责且 7 天未联系的客户

## 五、自动化流程
- **新客户提醒**：新客户录入 → 自动通知负责人
- **客户预警**：客户 30 天无跟进 → 推送预警到群

## 六、仪表盘
- **销售看板**：本月新增客户数、按状态饼图、未跟进客户列表

## 七、表单
- **客户登记表单**：5 个必填问题，可分享给非用户填写

## 八、角色权限
- **销售**：只能看自己负责的客户
- **管理员**：全部权限

## 九、使用场景演示
1. 销售小张登入：自动看到『我的待跟进』视图，10 个待跟进客户
2. 录入新客户：填客户登记表单 → 自动通知销售经理
3. ...

## 十、待你确认
- [ ] 整体架构是否符合预期？
- [ ] 有无需要新增/删除/调整的字段？
- [ ] 自动化触发条件是否合理？
```

用户批准 → 进 Stage 5。

## Stage 5 — 自动执行

**动作**：明确告诉用户『现在切到 lark-base skill 执行建表，详细日志写入 07_execution_log.md』然后：

1. 切到 lark-base skill
2. 按 [`execution_playbook.md`](execution_playbook.md) 编排顺序：base → tables → fields → views → roles → forms → workflows → dashboards
3. 每步用 `--dry-run` 先验证语法，然后真执行
4. 每步成功失败都写入 `07_execution_log.md`
5. 任何失败 → 立即停下 → best-effort 回滚已建对象 → 报告用户

**产出**：`07_execution_log.md`
```markdown
# 执行日志

## 2026-04-16 10:23:15 - +base-create
命令：lark-cli base +base-create --name "客户管理系统" --as user
结果：✅ 成功
返回：base_token=bascn..., url=https://...

## 2026-04-16 10:23:18 - +table-create (客户表)
...

## 失败回滚（如有）
...
```
