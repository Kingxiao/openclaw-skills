# 覆盖度审计：角色 × 场景 × 生命周期矩阵

> verified: 2026-04-16

## 为什么这是强制步骤（来自 2026-04-16 银杏家园实测暴露的缺陷）

历史教训：Stage 1 按"实体→关系→状态机→角色→场景"5 维度推理后，agent 直接写了 01_requirement.md，**跳过了角色×场景交叉审视**。结果 Stage 4 审批时用户发现 20+ 场景缺口：站点端没有月报提交表单、新媒体没有素材审核工作流、基金会没有脱敏视图定义、项目助理没有"我的待办"视图……

**根因**：5 维度是**并列**的，没有**交叉审视**——实体列得再全、角色定得再清，也可能漏掉"角色 A 在实体 B 上的日常操作 C"这种交叉场景。

**修复**：引入「角色 × 场景 × 生命周期」矩阵。Stage 1 必产出本矩阵的填空结果。Stage 3 必校验每个矩阵格都有对应 schema 组件支持。

---

## 矩阵模板

每个**角色**必须回答以下 6 类场景（CRUD-AV：Create / Read / Update / Delete / Approve / View）：

| 场景类别 | 核心问题 | 支撑组件 |
|---------|---------|---------|
| **C 录入** | 这个角色要录入什么数据？（通过 base 表单？飞书表？纸质？外部系统？） | Form / Table（直接录） |
| **R 查阅** | 这个角色日常要看什么数据？（全表？过滤后？聚合后？） | View / Dashboard |
| **U 更新** | 这个角色要修改什么字段？（哪些字段对他只读？哪些可改？） | Role（field_rule）|
| **D 删除** | 这个角色有无删除权？删除哪些？有无软删除/归档？ | Role（table_rule.perm）+ Status 字段 |
| **A 审批** | 这个角色在什么流程里作为审批人？上游谁给他推？他的决定推给下游谁？ | Workflow（LarkMessageAction 推送）|
| **V 通知** | 这个角色会收到什么自动提醒？（超期/新任务/风险告警）| Workflow（TimerTrigger / ChangeRecordTrigger）|

---

## 生命周期审查

对每个**核心业务对象**（不是每张表，是业务语义对象，如"客户/项目/老人"）问：

1. **产生**：从哪里来？谁录入？
2. **流转**：经过哪些状态？每个状态的触发条件？
3. **完成**：最终状态是什么？谁标记？
4. **归档/销毁**：是否永久保留？保留期限？撤回机制？

每个生命周期节点必须对应：
- 触发条件：哪个字段/哪个动作触发状态变化？（Field 或 Form 提交）
- 通知方：状态变化要通知谁？（Workflow LarkMessageAction）
- 可见性：状态变化后哪些角色能看到？（View / Role）

## 填空示例（CRM 场景）

### 角色：销售

| 场景 | 实现 |
|------|------|
| C 录入 | 客户登记表单 → 客户表 |
| C 录入 | 跟进记录：直接在客户详情页新增 |
| R 查阅 | 视图「我的客户」（filter: 负责人=ME） |
| R 查阅 | 视图「待跟进」（filter: 最近跟进<7天前） |
| U 更新 | 客户状态、跟进内容（field_rule） |
| D 删除 | 不可删客户（role: delete=false），可软删（状态=流失） |
| A 审批 | 无（销售不审批） |
| V 通知 | Workflow「7 天未跟进预警」推送 |

### 生命周期：客户对象

- **产生**：销售录入 OR 表单提交 → 状态=待联系
- **流转**：待联系 → 跟进中（销售改状态）→ 已签约（销售改状态+签约日）OR 流失（销售改状态）
- **完成**：已签约 / 流失
- **归档**：状态=流失 12 个月后自动归档（Workflow TimerTrigger）

## Stage 3 校验扩展

原 5 项 + schema_format_contract 新增 5 项之外，再加：

11. **角色场景覆盖**：每个 `roles[].role_name` 必须在 `01b_coverage_matrix.md` 的矩阵中出现，且每个 C/R/U/A/V 场景都有对应 schema 元素
12. **生命周期闭环**：每个核心业务对象的状态机从"产生"到"归档"必须闭合，每个状态转换必须有触发机制（字段改变 / 表单提交 / 定时 workflow）
13. **非飞书用户角色 I/O 闭合**：非飞书用户角色（如站点端、外部用户）的录入必须通过 Form，查阅必须通过分享的 Dashboard 或导出

## 常见遗漏模式（从银杏家园实测归纳）

| 遗漏模式 | 典型表现 | 防范 |
|---------|---------|------|
| **非飞书用户的上行通道不全** | 只建了"月度数据"表单，漏了"月度报告"/"整改反馈"/"授权书上传"等 | 审查：非飞书角色**所有 C 场景**是否都有 Form |
| **审批链多层漏中间层** | workflow 只推初审人，没推复核人/终审人 | 审查：每个 A 场景涉及几级审批，每级都有推送 |
| **动态权限未落到 record_rule** | 项目经理动态负责站点，但 role 里写死了 | 审查：动态关系用中间表 + `record_rule.filter` 驱动 |
| **"我的待办"视图缺失** | 角色权限有但没个人视图，日常要翻 | 审查：每个内部角色至少 1 个"filter = ME"的视图 |
| **脱敏视图只说"只读"没说"哪些字段可见"** | 基金会只读=看全部字段 | 审查：脱敏视图必须用 `field_rule` 明确"哪些字段不可见" |
| **状态机结束状态没归档逻辑** | 流失客户/完成项目永远堆在表里 | 审查：终态后是否有定时归档 workflow |

## 何时使用本文件

- **Stage 1 进行中**：写 01_requirement.md 时，每识别一个角色和一个业务对象，对照本模板填矩阵
- **Stage 1 结束前**：`01b_coverage_matrix.md` 必须产出（与 01_requirement.md 并列为 Stage 1 产出）
- **Stage 3 校验**：用矩阵表检查每个格子是否有 schema 元素支撑
- **Stage 4 审批**：06_proposal.md 按**角色维度**组织内容（不再按"表/视图/工作流"维度），每个角色一节展示他的完整故事
